Purpose
Ancilla is a personal LLM memory system. The core idea is not "chat with history", but an append-only personal event log that produces derived memories and assembles compact personal context on demand. The product goal is selective recall: always retrieve weak signals in the background, then inject only high-signal context into an LLM prompt when it is actually useful.

Core product model
- Immutable entries are the source of truth.
- Artifacts are derived from entries: transcripts, chunks, summaries, reflections.
- Memory records are derived, versioned, and temporally bounded.
- Retrieval is hybrid: lexical plus semantic plus deterministic policy features.
- Context assembly is selective and traceable.
- Chat is a client of memory, not the primary abstraction.

Current technology choices
- Backend: Rust
- Runtime storage: PostgreSQL when DATABASE_URL is set, local JSON fallback otherwise
- Vector support: pgvector in PostgreSQL
- Chat model: AWS Bedrock Converse when BEDROCK_CHAT_MODEL_ID is configured; deterministic synthetic fallback otherwise
- Embeddings: Perplexity 0.6B family
  - perplexity-ai/pplx-embed-v1-0.6b for query and memory embeddings
  - perplexity-ai/pplx-embed-context-v1-0.6b for artifact/chunk embeddings
- Local embedding execution: PyTorch, device auto-selection in this order: cuda -> mps -> cpu
- Infra provisioning: Terraform/OpenTofu

Key design decisions already reflected in the repo
- The backend no longer owns transcription or embedding generation. Clients/frontends can transcribe locally, compute embeddings locally, and submit transcript text, prepared artifacts, prepared memories, and query embeddings.
- Audio is represented as a source kind, but the backend does not need to run Transcribe for MVP.
- Embeddings are versioned/stored separately from memory rows and artifact rows.
- The system is designed so memory survives model swaps.
- Retrieval traces are persisted so injected context is explainable.

Current runtime behavior
- If DATABASE_URL is set:
  - the app runs migrations via sqlx
  - the app uses Postgres and the normalized schema
  - the retrieval path uses SQL candidate generation
- If DATABASE_URL is not set:
  - the app falls back to a local JSON state file
- If BEDROCK_CHAT_MODEL_ID is set:
  - POST /v1/chat/respond uses Bedrock Converse
- If BEDROCK_CHAT_MODEL_ID is not set:
  - chat uses the deterministic synthetic backend

Current infra shape
The infra was deliberately reduced from the more expensive baseline to something reasonable for a personal side project.

Current default deployment shape
- no ALB
- no NAT gateway
- ECS Fargate service in public subnets with a public IP
- ARM64 Fargate defaults: 256 CPU / 1024 MiB
- single-instance RDS PostgreSQL instead of Aurora
- S3 bucket
- ECR repository
- IAM roles/policies
- Secrets Manager secret
- optional network creation

Network shape when create_network = true
- 1 VPC
- 2 public subnets
- 2 reserved private app subnets
- 2 private DB subnets
- Internet Gateway
- route tables
- no NAT gateway by default

Why the infra was changed
- The earlier ALB + NAT + Aurora shape was too expensive for a personal side project.
- The repo now defaults to a cheaper stack that still keeps the database private while letting the app run publicly on ECS without a load balancer.
